Phase 2 — Architektur-Gate

Simplicity-First-Triage

StufeFrageAntwort
(a)Ganz ohne LLM?Nein. Transkript-Analyse → strukturierte Pro/Contra-Liste braucht LLM.
(b)Single LLM-Call, kein Agent?Nein. Router + Generator + Evaluator = 3 LLM-Calls. Aber: kein Agent-Loop, kein Tool-Use. Deterministischer Ablauf mit HITL-Gate.
(c)Multi-LLM-Pipeline mit HITL?Ja. Router (günstig) → Generator (stark) → Evaluator (mittel) → Slack HITL. Max 2 Regenerate-Loops.
Decision-Record:
Pattern: LLM-Augmented Pipeline mit HITL-Gate. Kein autonomer Agent. Drei LLM-Calls in fester Reihenfolge: Router klassifiziert, Generator produziert, Evaluator prüft. Mensch entscheidet via Slack-Bot (Bestätigen/Ablehnen/Editieren). Kein stiller Write nach Personio — jeder Write ist menschlich autorisiert.

Telefonnummer-Normalisierung

Ausgangslage: Close.com liefert Nummern mit +49-Präfix, aber die Formatierung variiert — mal mit Leerzeichen (+49 174 1234567), mal ohne (+491741234567), mal inkonsistent (+49 1741234567). Personio hat beliebige Formate, da Bewerber ihre Nummern selbst eingeben (0174..., 0049174..., +49 174..., +49174...).

function normalize(phone):
    stripped = remove_all(chars=' ()-/.\t', from=phone)
    
    if stripped starts with '00':
        stripped = '+' + stripped[2:]        // 0049... → +49...
    else if stripped starts with '0':
        stripped = '+49' + stripped[1:]       // 0174... → +49174...
    // else: already has + prefix, keep as-is
    
    return stripped  // +49174xxxxxxx (E.164, keine Trennzeichen)

Beide Seiten durchlaufen dieselbe Funktion → Match auf kanonischer Form. Die Normalisierung ist robust gegen alle Close.com-Formatierungsvarianten, da sie alle Nicht-Ziffern-Zeichen entfernt, bevor sie das Präfix normalisiert.
